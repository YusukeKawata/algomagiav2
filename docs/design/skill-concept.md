# スキルの概念（本作の根幹）

> 作成 2026-06-10 / ステータス: 正式（ADR-0002 §6 と一体。スキル設計は常にこの定義に則る）
> 実証コード: `experiments/skill_prediction.js`（オンライン予測の完全シミュレーション）

## 0. 定義（覚え書きと一対一対応）

- **スキルとは数学の関数である。** 1スキル ＝ 1つの関数 f(x)。
- **スキルの組み合わせで新しいスキルが生成できる**＝**関数と関数を組み合わせる**こと（和 f+g・積 f·g・合成 f∘g）。
- **攻撃は関数が生成した数列である。** x=0,1,2,… を順に代入した f(0),f(1),f(2),… が「弾の列」。
- **計算に必要なステップ数が、キャラが消費する魔素量**（攻撃側＝数列生成の素演算数）。
- **関数には様々な種類が存在する**（多項式・べき乗・周期・漸化式・剰余…＝検討済みの族）。
- **関数を組み合わせるほど数列は複雑になり、相手は元の関数を推測しにくくなる。**
- **防御側は数列を前から1つずつ確認し、次の数を予測する。** 予測できればダメージ無し、できなければ被弾。
- **防御側も予測の計算ステップ1つにつき魔素1を消費する。**

補足の原則（ADR-0002）:
- ダメージは値の**大きさ(magnitude)では決まらない**。決めるのは**構造（どんな関数か＝予測しにくさ）**。
- 最適化は出力 f を変えず、**生成/予測の計算を安くする**こと（Horner 等）。

## 1. 防御側の予測アルゴリズム（オンライン有限差分）

防御側は数列を前から受け取り、**有限差分の対角線**を保ちながら「次の数 ＝ 差分対角線の総和」で外挿する。
- k階差分が 0 になったら「次数 k−1 の多項式だ」と**看破（ロック）**し、以降は正確に予測＝無傷。
- 多項式でない数列（和に周期などの異族が混ざる）は**差分が 0 にならず**、看破できず**外し続ける＝被弾**。
- 予測1手・差分1回がそれぞれ魔素1。看破に必要な手数が防御側の消費魔素。

これは「より良い予測器（＝看破モデル＝耐性）を持つほど安く看破できる」という §3 の**耐性＝看破モデル**と一体。
多項式予測器を持つ防御は多項式を看破できるが、周期混じりは看破できない（周期予測器という別の耐性が要る）。

### 1-1. 防御の計算ステップと消費魔素（実トレース）

**予測＝差分対角線の総和（加算）／更新＝受信値から差分を取り直す（引き算）**。各素演算1回＝魔素1。
**防御魔素はこのステップ数の総和**。`experiments/skill_prediction.js` が各項のステップを印字する。

**① 1次 `3+2x`**（3項目で看破→以降ロックして無傷。合計49魔素）
```
n=1 受信5 ｜ 予測 3=3       → 外被弾 ｜ 更新 5−3=2            →[5,2]    ｜ 予測0+更新1=1 ｜累計1
n=2 受信7 ｜ 予測 5+2=7     → 当無傷 ｜ 更新 7−5=2, 2−2=0      →[7,2,0]★2階差分=0→1次と看破 ｜ 予測1+更新2=3 ｜累計4
n=3 受信9 ｜ 予測 7+2+0=9   → 当無傷 ｜ 更新 9−7=2,2−2=0,0−0=0 →[9,2,0]  ｜ 予測2+更新3=5 ｜累計9
…以降 毎項 予測2+更新3=5魔素 → 合計49（看破まで4 ＋ 5×9項）
```

**⑤ 2次＋周期2（非多項式）**（差分が0にならず看破できず全弾被弾。合計101魔素）
```
n=3 受信15 ｜ 予測 5+(-2)+(-8)=-5  → 外被弾 ｜ 更新 15−5=10,10−(-2)=12,12−(-8)=20 →[15,10,12,20]
n=4 受信17 ｜ 予測 15+10+12+20=57  → 外被弾 ｜ 更新 …                          →[17,2,-8,-20,-40]
…差分が0にならず対角線がK(=6)まで伸び、毎項13魔素(予測6+更新7)を浪費しながら外し続ける → 合計101・被弾12
```

要点: 看破できれば**以降ノーダメージ**（一定魔素で予測が当たる）、できなければ**毎項より多くの魔素を使い外し続けて被弾**。
③(3次)は n=4 で看破→以降 予測4+更新5=9魔素/項、というように**次数が上がるほど看破後の1項魔素も増える**。

## 2. 実証（`experiments/skill_prediction.js` の結果）

単純な関数→合成した関数へと移したときの、攻撃側生成魔素・生成数列・防御の結論・防御魔素・被ダメージ:

| スキル(関数の組み合わせ) | 展開した式 | 1項魔素 素直→Horner簡約 | 数列(先頭6) | 防御の結論 | 防御魔素 | 被弾 |
|---|---|---|---|---|---:|---:|
| ① 1次 | `3 + 2x` | 2→2 | 3,5,7,9,11,13 | 1次と看破 | 49 | 2 |
| ② 2次 | `1 + x²` | 4→4 | 1,2,5,10,17,26 | 2次と看破 | 65 | 3 |
| ③ 1次×2次=3次 | `2x³+3x²+2x+3` | 7→6 | 3,10,35,90,187,338 | 3次と看破 | 79 | 4 |
| ④ 2次×2次=4次 | `x⁴+2x²+1` | 9→8 | 1,4,25,100,289,676 | 4次と看破 | 91 | 5 |
| ⑤ 2次＋周期2 | `(x²+1)+[0,5,0,5,…]`（非多項式） | 6→— | 1,7,5,15,17,31 | **看破失敗** | 101 | **12(全弾)** |

読み:
- 攻撃魔素は「1項魔素 × 項数(L=12)」。例 ① は 2魔素/項 × 12 = 24。
- **積で合成**すると次数が上がり、防御は看破できるが**手数(防御魔素)も被弾も漸増**（①→④で 2→3→4→5）。
- **和で異族(周期)を合成**すると**非多項式**になり、多項式予測器は**原理的に外し続ける**＝全弾被弾・防御魔素最大。
- 防御の予測過程（差分表が1項ずつ進み、①は3項目で看破→以降無傷、⑤は 1→13→-5→57… と外し続ける）も
  同スクリプトの詳細トレースで確認できる。

### 最適化の余地（②の論点）
「1項魔素 素直→Horner簡約」が示すとおり、**合成は最適化の余地を生む**:
- ③ は素直に評価（各因子を計算して掛ける）で7手、大賢者が**展開してHorner評価**すると6手（④は9→8）。
- 一方 ① `2x+3` や `x²−1=(x+1)(x−1)` のような**最小形は余地なし**（2→2）。これは正常で、
  そのときは強さを**最適化ではなく構造（次数＝予測されにくさ）**で稼ぐ。
- 大賢者の最適化＝**合成式を代数的に簡約して最小手数にすること**（展開・因数分解・Horner）。
  最適化は「同じ魔素で多く撃つ」レバー、構造（合成での次数・非多項式化）は「看破されにくくする」レバー。

→ **スキル合成が「予測されにくい関数を作る」**ことを、攻守の魔素・数列・予測アルゴリズム・被弾で実証した
（以前のような“モック”ではない）。

## 3. ゲーム本体との関係（実装：本物のシミュレーションで戦う）

- **戦闘の防御はこの実シミュレーションそのもの**。`src/core/mechanisms/predictor.ts` の `defendSim` が、
  攻撃ロードアウトの**実数列**（各スキルの数列の和）を生成し、防御が**前から1項ずつオンライン予測**する。
  ダメージ＝予測を外した項数、防御魔素＝予測ステップ数。`m1.ts` の `resolveAttack` がこれを呼ぶ
  （以前の“早見式”は廃止し、本物の予測に置換）。`predictor.test.ts` で挙動を固定。
- 各スキルは実数列を生成する（`skillSequence`）: 多項式(係数)/周期(ブロック)/漸化式(線形漸化)/べき乗。
  **統合＝関数の和**でロードアウトを1つの数列に合成（数列が実際に複雑化）。
- 防御の耐性 = 看破モデル: 周期/漸化式の成分は対応耐性があれば**除去(看破)**し、残差を
  多項式予測器（次数上限＝多項式耐性）で予測。1つでも未看破の非多項式成分が残れば外し続ける＝全弾命中。
- スキル構成 `skill.ts` は ADR-0002 §6 に従い関数を**実際に組み立てる**
  （業火: 種火=低次係数 / 増幅=高次係数で次数↑ / 収束=評価）。
- **dev🐞パネル**（`npm run dev`）は、現在の攻撃 vs 現在の敵耐性で `defendSim` を走らせ、
  攻撃数列・除去した成分・各項の予測/判定/項魔素・合計防御魔素・被弾を表示する（戦闘と同じ計算）。
- 積・合成による次数上げ（係数の畳み込み）は今後 統合操作として展開する。

## 4. 今後スキルを設計するときの遵守事項

1. スキルは関数として定義する（族・次数/周期/階数・係数）。値の大きさは強さにしない。
2. 合成は関数演算（和/積/合成）として実装し、**数列が実際に複雑化**することを `skill_prediction` 系の
   シミュレーションで確認する（防御魔素・被弾が動くこと）。
3. 防御の看破は族ごとの予測器（耐性）。新しい族を足すなら、対応する予測器(耐性)も設計する。
4. 最適化は評価/生成の計算を安くするもの（出力不変）。
