# UX設計: ノードグラフ式スキル工房（最適化の操作系）

> 作成日: 2026-06-09 / ステータス: ドラフト
> 対応プロトタイプ: `experiments/node_optimize.js`（エンジン部は実装・検証済み）

プレイヤーが「最適化」を**ノードの編集**として行うUX。エンジン（魔素計量と出力検証）は
`node_optimize.js` で動作実証済み。本書はその上に乗せる操作系の設計。

---

## 1. 基本思想

- **スキル＝演算ノードのデータフローグラフ**（種火→増幅→収束→着弾）。
- **魔素＝各ノードが実行する素演算の総数**（決定論的・実測）。
- **最適化＝ノードの書き換え**。出力を変えずに魔素を下げる。
- ゲームが**出力の同一性を自動検証**するので、「等価でない改変」はそもそも適用不可
  （プレイヤーはズルできない／壊せない）。＝安全に編集できる。

---

## 2. コアループ（原作トレース）への接続

```
 戦闘で敵を倒す
   └► [解析] 敵スキルのノードグラフを鹵獲（未最適化の状態で手に入る）
        └► [工房] ノードを編集して魔素を下げる（＝本書のUX）
             └► [装備] 最適化したスキルをセット
                  └► 次の戦闘: 同じ演算ゲージでも、安くなった分だけ多く撃てる＝強い
```
（戦闘側の実証は `combat_loop.js`：Horner習得で同ゲージのままゴーレムに勝てるようになる）

## 4. 操作（インタラクション）

| 操作 | 内容 |
|---|---|
| ノード選択 | グラフ上のノードをクリック/カーソル。詳細（種類・現状の方式・魔素）を表示 |
| 規則適用 | 習得済みの「最適化規則（道具）」を選択ノードに適用。プレビューで before/after 魔素を表示 |
| 自動検証 | 適用前にゲームが「出力が一致するか」を全入力で検査。不一致なら適用不可（赤表示） |
| 取消/履歴 | Undo/Redo。試行錯誤を奨励（壊れないので安心して触れる） |
| 結線編集(上級) | ノードの組み替え・共通部分の括り出し（CSE）。スキル統合の核 |

設計指針:
- **失敗しても壊れない**（等価でない編集は弾く）ので、初心者は安心して触れる。
- **プレビューで魔素差分を即表示** → 「効いてる感」を可視化（飾りでなく実数）。
- 上級者は結線まで編集して**スキル統合**（複数スキルを1グラフに融合）まで踏み込める。

## 5. 最適化規則カタログ（＝プレイヤーが習得していく「道具」）

`node_optimize.js` で実装・検証済みの3つ＋拡張余地。

| 規則 | 対象ノード | 効果（魔素） | 元ネタ | 習得元の例 |
|---|---|---|---|---|
| 強度低減 | buildAdd | +1をN回 → ×1回（N→1） | skill_loop 第1幕 | スライム |
| 高速累乗 | pow | 逐次 N-1回 → O(log N)（1024→12） | 第2幕 | 魔術師 |
| Horner融合 | poly | O(d²) → O(d)（54→16） | 第3幕 | 賢者 |
| （拡張）CSE | 任意 | 重複計算の括り出し | 共通部分式除去 | — |
| （拡張）メモ化 | 再帰 | 同入力の再計算を省く | memoization | — |

> 各規則は「敵スキルの解析」や「クエスト」で習得する＝**道具が増えるほど、より深く最適化できる**。
> これが「大賢者の能力が育つ」体験に対応する。

---

## 6. 難易度・段階づけ

- **序盤**: 規則は「ワンタップ適用」（ノードを選んでボタン）。中身を理解しなくても遊べる。
- **中盤**: どのノードにどの規則が効くかを自分で見極める（パズル性）。
- **終盤/上級**: 結線編集・スキル統合・自作規則（上級者向けに、実際に小さな式を書ける“賢者モード”も検討）。

UX方針の要点: **「コードを書かせる」ことを必須にしない**。既知規則の適用＝ノード操作を主動線にし、
書きたい人だけが踏み込める二段構え。これでプログラミング素養がなくても遊べ、ある人は深く潜れる。
